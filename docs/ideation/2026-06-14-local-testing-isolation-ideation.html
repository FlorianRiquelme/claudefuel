Local testing without polluting your daily-driver claudefuel

Ideation · 2026-06-14 · repo-grounded · focus: a way to test claudefuel changes locally without disturbing the released version you use every day

Framing

The grounding surfaced an important distinction the request blurs: automated testing is already isolated. The bats harness sets CLAUDE_CONFIG_DIR="$TMP" (a mktemp dir) and mocks the network through CLAUDEFUEL_API_BASE / CLAUDEFUEL_RAW_BASE, so bats tests/ never touches your real install.

The unmet need is interactive testing — actually seeing a dev build render in a live bar, or exercising the install / claudefuel update flow end-to-end. Today both require touching a real CLAUDE_CONFIG_DIR, and the default is ~/.claude: your daily driver, running a released version. Pollution here is two-faced:

  • File pollution — dev builds of statusline.sh, the claudefuel binary, and the five slash commands overwrite your stable copies and globally re-register dev-version commands.
  • State pollution — bumping the # claudefuel: vX.Y.Z header to test makes /claudefuel.update refuse with "installed is newer than the latest release" (README troubleshooting #194–195). The real install's update path is left wedged.

One structural gap enables both: claudefuel update only reconciles from upstream GitHub (release-tag-pinned). There is no first-class way to reconcile from the working tree, so the real codepath can't be driven against your branch without hand-copying files.

Every idea below is weighed against the project's own Maintenance ceiling (CONTEXT.md): every code path is one the solo maintainer must test, so convention beats machinery wherever it can.

Grounding context

Project shape: a Claude Code status bar (bash + jq) self-installed into ~/.claude/: statusline.sh, the claudefuel binary, commands/claudefuel.*.md, and cache/. The maintainer both develops and daily-uses it.

Existing levers: TARGET_DIR="${CLAUDE_CONFIG_DIR:-$HOME/.claude}" (install respects the env var); API_BASE/RAW_BASE overrides (test-only today); tests/fixtures/; inspect_state JSON; setup.sh profile creation (built for multi-account, not dev sandboxing).

Topic axes (the surface these ideas cover):

  • Target isolation — where a dev build installs
  • Build source — getting the working tree into the sandbox
  • Live rendering — seeing the bar without a real session
  • Lifecycle — create / reset / tear down cleanly
  • Guard friction — version header & drift signal during dev

Survivors

Ordered by cost-to-leverage. #1 is the do-first; #2–#3 build the real end-to-end path; #4–#5 remove the live-session dependency.

1. Sandbox-by-convention + a real-profile write guard lowest cost

Adopt CLAUDE_CONFIG_DIR=~/.claude-cfdev as the documented dev convention, and add one small guard so dev helpers refuse to write when the resolved target is the real ~/.claude. Almost no new machinery — it formalizes a lever that already exists and makes the safe path the default path.

  • target isolation
  • guard friction
Basis
direct: TARGET_DIR="${CLAUDE_CONFIG_DIR:-$HOME/.claude}" (claudefuel:13) already keys off the env var; setup.sh already builds sibling profile dirs.
Why it matters
Solves ~80% of the pollution problem with documentation plus a few lines of guard. Honors the Maintenance ceiling better than any code-heavy option.
Meeting test
"Should the dev workflow be a convention we document, or a feature we build?" is exactly the call worth making before writing code.

2. --from <local dir> reconcile source enabling primitive

Let claudefuel update reconcile from a local directory (or file://) instead of only upstream GitHub. The tests already override RAW_BASE/API_BASE for exactly this; promote it to a first-class, human-facing flag. This is the smallest independently-shippable change that makes the real update/install codepath testable against your working tree.

  • build source
  • guard friction
Basis
direct: API_BASE="${CLAUDEFUEL_API_BASE:-...}" / RAW_BASE="${CLAUDEFUEL_RAW_BASE:-...}" (claudefuel:16–17) are described in-file as "Overridable for tests."
Why it matters
Without this, "test locally" means hand-copying files — which diverges from the path users actually run. A local source closes the test/prod gap on the riskiest flow.
Meeting test
Exposing an internal test seam as a public flag has contract implications (CONTEXT.md stability contract) — worth a deliberate decision.

3. claudefuel sandbox — one-command editable install full workflow

A single subcommand that (a) targets a throwaway profile, (b) installs the working-tree bundle via #2, optionally symlinking statusline.sh so edits are live with no reinstall (like npm link / pip install -e), (c) prints the CLAUDE_CONFIG_DIR=… claude launch line, and (d) supports --ephemeral teardown. Built on #1 and #2.

  • target isolation
  • build source
  • lifecycle
Basis
reasoned: composes the two existing levers (env-var target + overridable source) into the workflow the maintainer repeats on every manual test. external: editable-install precedent (npm link, pip -e).
Why it matters
Turns a multi-step ritual into one command, reused forever — compounding leverage. The symlink mode collapses the edit→see loop to zero reinstall.
Meeting test
This is a new public verb on a tool whose whole identity is minimalism — does it earn its place against the Maintenance ceiling, or is #1+#2 enough?

4. claudefuel preview — render the bar from fixtures, no session no usage burn

Pipe Claude Code's real statusline stdin JSON contract from tests/fixtures/ into statusline.sh and print the three lines — across a state matrix (low/med/high usage, drift on/off, cap-ETA, multi-profile). No live Claude session, no API calls, no real usage burned. Think Storybook for the bar.

  • live rendering
Basis
direct: tests/fixtures/ already exists and feeds bats; the bar's input is a documented stdin JSON contract.
Why it matters
Removes the live-session dependency entirely — the fastest, cheapest, most deterministic way to eyeball rendering changes, and it never touches any profile.
Meeting test
Worth discussing whether this replaces most need for the sandbox (#3) for pure rendering work, narrowing #3's justification.

5. Promote tests/fixtures/ to the shared rendering contract compounding

Make one fixture set the single source of truth feeding both bats and preview (#4). Every new edge case added for an automated test instantly becomes a visual preview state, and vice-versa — coverage compounds instead of forking.

  • live rendering
  • lifecycle
Basis
direct: fixtures already drive bats. reasoned: a shared corpus prevents drift between "what we test" and "what we eyeball."
Why it matters
Low effort, high half-life. Mostly an organizing decision rather than new code.
Meeting test
A convention worth agreeing on before fixtures multiply in two incompatible shapes.

Rejected (with reasons)

IdeaWhy it didn't survive
Containerize a throwaway $HOME / devcontainer so even ~/.claude is sandboxed Maintenance-ceiling violation. Disproportionate weight for a bash+jq tool whose entire value proposition is staying small; CLAUDE_CONFIG_DIR already gives 100% of the isolation a container would.
Git worktree per dev build Isolates source branches, not the install — orthogonal to the stated pollution problem. (Repo already has .claude/worktrees for the branch-isolation need.)
Ephemeral self-deleting sandbox as a standalone feature Real but minor; folded into #3 as a --ephemeral flag rather than its own surface.
"Document it in the README" as a distinct idea Not an idea — it's the delivery vehicle for #1. Folded in.
