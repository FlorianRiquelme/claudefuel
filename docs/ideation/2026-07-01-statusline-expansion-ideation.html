Codebase Context

claudefuel is a pure bash+jq+curl statusline for Claude Code (~680 lines in statusline.sh), distributed as a "Promptfile" (ADR-0001) rather than a plugin. Display: Line 1 [profile] Model | ctx ●●○○ used/total | thinking | effort | ↗ update-drift; Line 2 5h/7d/extra bars + prepaid balance; Line 3 reset times + cap-ETA. Rendering is inline string concatenation, so segment show/hide/reordering is structurally impossible without the config-loader work in feat/configure-foundation. build_bar's hue thresholds (red≥90/yellow≥70/orange≥50/green) are the only severity channel. Line2/line3 silently collapse to empty strings on fetch failure with no cache — a gap feat/honest-instrument already addresses. The render path makes 5 sequential curl calls (~23s worst case) with a CLAUDEFUEL_OFFLINE gap that feat/never-block-render already closes. Multi-account isolation uses a CACHE_SUFFIX hash per CLAUDE_CONFIG_DIR; all profile caches land in one flat, globbable /tmp/claude/ directory — feat/cross-profile-headroom already reads across them via a new /claudefuel.fleet skill. claudefuel_cap_eta_segment computes elapsed/pct/pace as a pure function of one snapshot (ADR-0004: no cross-render state, no daemon) — feat/burn-radar already extends this into a pace chip and steer-to framing. input_tokens/cache_create/cache_read are fetched but only shown as a collapsed sum — feat/conversational-copilot's /claudefuel.why already exposes richer detail over a versioned snapshot API. /claudefuel.configure is a shipped, documented stub with zero config keys wired until feat/configure-foundation lands. No $COLUMNS/terminal-width awareness exists anywhere. No timing instrumentation exists in /claudefuel.doctor on any branch, including the performance-focused ones.

Hard constraints: ADR-0003 (bash+jq+curl only, no daemon/plugin runtime/TUI editor; customization limited to thresholds/ordering/show-hide/presets), ADR-0004 (predictions must be pure functions of one snapshot), and a named "cliffs" taxonomy of 7 triggers that would force a rewrite. External research found direct competitors (ccstatusline, Claude-Code-Usage-Monitor) already shipping historical charts, per-number provenance labeling, and Sonnet/Opus split — and confirmed user sentiment ("flying blind," "I built a status light for Claude Code" — 3.4k upvotes) strongly favors ambient, glanceable, honest indicators over raw numbers.

/claudefuel.doctor Latency Instrumentation

Render mechanics & performance

Add per-fetch timing (drift / OAuth refresh / usage / account / prepaid) to /claudefuel.doctor, so the render-path work already done on feat/never-block-render has a regression gate. Right now nothing on any branch measures render latency — the ~23s worst-case figure this session confirmed is only known from reading timeouts, never from an actual timed run.

Basis
direct: confirmed via search — no timing/latency instrumentation exists in claudefuel.doctor.md on this branch or on any of the 7 sibling feature branches.
Rationale
Without this, the performance fix in feat/never-block-render has no before/after measurement and future regressions are undetectable — it's the missing feedback loop that makes that branch's claims verifiable rather than anecdotal.
Downsides
Adds a small amount of doctor-only code; needs to avoid adding runtime cost to the hot render path itself (all instrumentation should live in the diagnostic surface, not the per-prompt render).

High — 85%

Config Self-Report for Support at Scale

Render mechanics & performance (diagnostics)

Once feat/configure-foundation's config loader ships, extend /claudefuel.doctor to emit one copy-pasteable line — effective config, version, profile hash — designed for pasting into a GitHub issue. As soon as real config keys exist, "what config is actually in effect on this user's machine" becomes an unanswerable-by-the-maintainer question without it.

Basis
reasoned: claudefuel.json is user-owned and never touched by install/update — the moment /claudefuel.configure ships its first real key (via feat/configure-foundation), config drift becomes a live support cost with no diagnostic surface today.
Rationale
Cheap to build now, expensive to retrofit once users have diverse configs in the wild and bug reports start arriving without enough context to reproduce them.
Downsides
Sequencing matters — there's nothing to report until feat/configure-foundation actually merges, so this should ship alongside or just after it, not before.
Confidence
Medium-High — 75%
Complexity
Small

Severity Beyond Hue (WCAG-safe glyph pairing)

Alarm & failure-state design

feat/calm-cockpit's build_bar was checked directly and still resolves severity through bar_color alone — hue is still the only severity channel even on the branch named for "progressive alarm as structure, not hue." Pair each severity tier with a fixed glyph or shape (not just color) so severity survives colorblindness and inconsistent terminal color profiles.

Basis
direct: read feat/calm-cockpit's own diff — bar_color assignment (red/yellow/orange/green) is unchanged from main; no glyph or shape pairing was added. external: WCAG 1.4.1 requires color never be the sole signal.
Rationale
This is a real, currently-unaddressed accessibility gap even after crediting the branch whose name suggests it should already be fixed — worth flagging precisely because the branch name is misleading about what it actually delivers.
Downsides
Frame this as a refinement PR on top of feat/calm-cockpit, not a new standalone feature — building it in isolation would conflict with that branch's changes to the same function.

Medium — 65%

Rate-of-Change Severity

Alarm & failure-state design

None of the 7 branches address velocity: an account sitting steady at 40% and one that just jumped 15 points in the last render look visually identical today, because build_bar's hue is purely a function of absolute percentage. This is presented as a flagged architectural question, not a ready-to-build feature.

Basis
reasoned: computing rate-of-change requires comparing the current snapshot against a prior one — a form of cross-render state ADR-0004 currently forbids for predictive features. ADR-0004 also explicitly names "instantaneous burn rate" as an already-considered-and-rejected idea for exactly this reason.
Rationale
The blind spot is real, but building it without an explicit ADR ruling would silently re-open a question the team already closed once. Worth surfacing as a deliberate "should we grant a narrow, scoped exception" decision, not as a backlog item to just build.
Downsides
Architecturally contested by design — do not implement without first getting an explicit ADR-0004 amendment or a documented exception.
Confidence
Low-Medium — 40%
Complexity
Medium
ADR-0004 tension — do not build without an explicit team ruling on the no-cross-render-state constraint.
